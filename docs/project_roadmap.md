# MedicalGPT 项目总路线图

项目名：`MedReason-Align`

目标：基于 `MedicalGPT` 做一个“结构化病例诊断推理”项目，通过“目标分布对齐的数据构造 + SFT + DPO + 严格评测”完成闭环，而不是只跑通一个训练脚本。

## 1. 最终要做成什么

你最终要交付的不是“一个会聊天的医疗模型”，而是：

- 输入：结构化或半结构化病例描述
- 输出：主诊断、依据、鉴别诊断、下一步检查、风险提示
- 形式：尽量稳定输出为 JSON 或固定模板
- 证据：公共 benchmark + 自建 holdout 集都能看到提升

### 项目定位

- 主任务：结构化病例诊断推理（Clinical Case Reasoning）
- 训练路线：`Target-aligned SFT -> DPO`
- 可选增强：`OPD`
- 暂不优先：`PT / PPO / GRPO`

## 2. 为什么这样设计

核心思想来自三件事：

- `HealthAI-2025`：先定义目标分布，再筛数据、做蒸馏
- `MedicalGPT`：提供完整的训练底座
- `lm-evaluation-harness`：提供统一的外部评测入口

你要证明的不是“我用了某个模型”，而是：

> 我通过更合理的数据构造和后训练，让模型在目标任务上真的变强了。

## 3. 数据方案

### 3.1 数据分层

| 层级 | 作用 | 是否进训练 |
|---|---|---|
| 原始候选池 | 提供广覆盖原始样本 | 是/否都可能 |
| 目标分布集 | 定义“你想让模型擅长什么” | 否 |
| SFT 训练集 | 教模型学会结构化病例推理 | 是 |
| DPO 偏好集 | 让模型偏向更好的回答 | 是 |
| Final Holdout | 最终盲评 | 否 |

### 3.2 推荐训练数据来源

优先级建议如下：

1. `FreedomIntelligence/HuatuoGPT-sft-data-v1`
   - 适合作为中文医疗指令候选池
2. `shibing624/medical`
   - 适合作为大规模医疗语料与指令候选池
3. `FreedomIntelligence/CMB`
   - 适合作为医学考试/临床推理风格数据
4. 少量 `GBaker/MedQA-USMLE-4-options-hf`
   - 仅做英文迁移补充，不要喧宾夺主

### 3.3 目标分布集怎么做

单独做一份 `target-dev-distribution set`，建议 200~500 条，全部是你想要的任务风格：

- 病例描述
- 主诊断
- 依据
- 鉴别诊断
- 下一步建议
- 风险提示

这份数据不参与训练，只用于：

- 定义目标任务分布
- 做相似度筛选
- 检查数据是否“长得像目标任务”

### 3.4 SFT 训练集怎么组成

建议配比：

- 60%~70%：目标病例推理样本
- 20%~30%：通用医疗问答样本
- 10%：安全边界样本

建议规模：

- 第一版：8k~15k
- 高质量优先，不要一开始追求大规模

### 3.5 DPO 偏好集怎么做

建议 2k~5k 条，每条包含：

- `prompt`
- `chosen`
- `rejected`

`chosen` 应该：

- 更完整
- 更保守
- 更有依据
- 更注重风险提示

`rejected` 应该故意包含：

- 漏关键依据
- 过度下结论
- 缺少鉴别诊断
- 缺少风险提示
- 格式不合规

## 4. 评测方案

### 4.1 公共 benchmark

建议使用：

- `CMB-Exam`
- `CMB-Clin`
- 可选：`MedQA-USMLE-4-options-hf`

用途：

- 给项目一个外部参照
- 证明你不是只在自建数据上“自嗨”

### 4.2 自建内部评测

建议单独做 300~500 条 `Final Holdout`，覆盖：

- 呼吸
- 消化
- 心血管
- 神经
- 内分泌
- 急诊

每条样本最好带标签：

- 科室
- 难度
- 是否高风险
- 是否需要鉴别诊断
- 是否需要结构化输出

### 4.3 评测指标

主指标：

- 主诊断准确率
- Top-k 命中率
- 鉴别诊断覆盖率

结构化指标：

- JSON 合法率
- 字段完整率
- Schema 匹配率

安全指标：

- 风险提示召回率
- 过度自信率

文本质量：

- 依据完整性
- 推荐可用性

### 4.4 评测原则

必须遵守：

- 训练集、筛选集、最终评测集严格隔离
- 公开 benchmark 和内部 holdout 都要报
- 做 ablation，不要只报一个总分
- 记录失败案例，不要只展示成功案例

## 5. 模型与训练策略

### 5.1 推荐 base model

建议主实验用：

- `Qwen/Qwen3.5-2B`

如果算力紧张，可先用：

- `Qwen/Qwen3.5-0.8B`

### 5.2 训练主线

第一版最合理的路线：

1. `Baseline SFT`
2. `Target-aligned SFT`
3. `Target-aligned SFT + DPO`

可选增强：

- `OPD`

不建议第一版就做：

- `PT`
- `PPO`
- `GRPO`

原因：

- 你现在最缺的是数据闭环和评测闭环，不是训练花样

## 6. 21 天执行计划

### Week 1：定任务、定数据、定评测

| 天数 | 产出 | 你要面对的问题 | 完成标准 |
|---|---|---|---|
| Day 1 | 项目定义 v1，任务边界写清 | 任务太宽，容易散 | 明确主任务只做病例推理，不做泛聊天 |
| Day 2 | 数据源清单 + 质量标准 | 公开数据风格不统一 | 列出每个数据集用途、风险、是否可训练 |
| Day 3 | 目标分布集设计稿 | 不知道“目标样本长什么样” | 写出 200~500 条样例的字段规范 |
| Day 4 | 数据 schema 定稿 | 问答、病例、考试题格式不同 | 所有样本可映射到统一 schema |
| Day 5 | 评测方案草案 | benchmark 选错会偏题 | 公共 benchmark + 自建 holdout 同时定下 |
| Day 6 | 数据过滤规则 | 噪声太多、泄漏风险高 | 写出去重、去泄漏、去低质规则 |
| Day 7 | 第一版项目说明文档 | 容易写成空话 | 文档里能回答“做什么、怎么做、评什么” |

### Week 2：做数据、做 baseline

| 天数 | 产出 | 你要面对的问题 | 完成标准 |
|---|---|---|---|
| Day 8 | 原始候选池整理完成 | 数据分散、字段不一致 | 候选池合并成统一格式 |
| Day 9 | 相似度筛选脚本 | 目标分布不稳定 | 能从候选池筛出目标任务相似样本 |
| Day 10 | 第一版 SFT 数据 | teacher 改写容易幻觉 | 至少能产出几千条可训数据 |
| Day 11 | SFT baseline 跑通 | 训练参数和格式容易错 | 能完整训出一个 baseline checkpoint |
| Day 12 | baseline 快速评测 | 不知道提升是否真实 | 公开 benchmark + 内部样本都能跑通 |
| Day 13 | 失败样本分析 | 模型会漏诊、乱答、格式差 | 归纳出至少 3 类高频错误 |
| Day 14 | 数据修订版 v2 | 数据质量不够稳定 | 修正后能明显减少明显坏样本 |

### Week 3：DPO、对比实验、定版

| 天数 | 产出 | 你要面对的问题 | 完成标准 |
|---|---|---|---|
| Day 15 | DPO 偏好集 v1 | chosen/rejected 难构造 | 每条样本能说清楚偏好原因 |
| Day 16 | DPO 训练跑通 | 训练不稳定、显存压力 | 至少产出一个可用 DPO checkpoint |
| Day 17 | SFT vs DPO 对比 | 不知道改进来自哪里 | 输出统一对比表 |
| Day 18 | 内部 holdout 最终评测 | 评测集太像训练集 | 只用未参与构造的数据做最终测试 |
| Day 19 | 误差分析报告 | 改进点不明确 | 分类出失败原因：漏信息、过度自信、格式错误等 |
| Day 20 | Demo / API / 页面展示 | 训练成果不好展示 | 能演示 before/after 对比 |
| Day 21 | 最终项目报告 | 容易变成流水账 | 有任务、数据、训练、评测、案例、结论 |

## 6.5 当前进度更新 (截至 Day 10)

### 已完成内容
- **Day 10**: 使用 DeepSeek V4 模型教师重写了 1000 条数据，用于第一版 SFT 数据构造。
- **基座模型基准**: 使用 Qwen2.5-7B-Instruct 在 CEval benchmark 上测试，整体准确率 0.7949 (acc 和 acc_norm)。
- **微调训练**: 在重写后的 1000 条数据上进行 SFT 训练，使用 Qwen2.5-7B 结构模型。
  - 训练损失: 0.8291
  - 评估损失: 0.7307
  - 困惑度: 2.0766
  - 训练样本: 737 (实际使用样本数)
  - 训练时间: 约 7 分 51 秒
  - 模型保存至: outputs-sft-medreason-qwen25-7b-1k
- **评测结果**: 微调后模型在 CEval-valid 上准确率 0.8009，提升 0.006 (约0.6%)。部分科目如 college_programming (0.8378 vs 0.8108)、legal_professional (0.6957 vs 0.6522) 有提升，但 advanced_mathematics (0.3684) 等仍低。

### 下一阶段计划
- 继续 Day 11-12: 完成 SFT baseline 训练并进行快速评测。
- 评估微调后的模型在 CEval 和内部 holdout 上的表现，与基座模型对比。
- 准备进入 DPO 阶段，构造偏好数据集。

### 备注
- 实际使用的基座模型为 Qwen2.5-7B-Instruct (计划推荐 3.5-2B，但因算力允许使用更大模型)。
- 数据规模为 1000 条重写样本，符合计划中的“至少能产出几千条可训数据”目标 (当前为 1k，后续可扩展)。

## 8. 你每天必须盯住的风险

- 数据泄漏：final eval 绝不能参与筛选
- 任务漂移：不要从病例推理跑成泛聊天
- teacher 幻觉：重写数据要做人工抽查
- 类别不均衡：不同科室要尽量覆盖
- 评测失真：不要只看一个 benchmark
- 结论过度：医疗场景必须保守输出
- 算力不足：先做 `SFT + DPO`，别先碰复杂 RL
